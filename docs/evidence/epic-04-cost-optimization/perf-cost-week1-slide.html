AWS us-east-1: VPC, ALB, EKS, NAT Gateway

Inbound: User → IGW → ALB → frontend-proxy → EKS  |  Outbound: EKS → NAT → ECR. Single NAT Gateway — trade-off trong Week 1.

Trade-off được ghi nhận

3 bottleneck đã xác định từ trace và code review

Cả 3 giải pháp dùng hạ tầng hiện có — không thêm managed service mới, không tăng cost trong Week 1.

B1 N+1 Currency Conversion
Vấn đề: Product listing với currency khác USD gọi currency gRPC theo từng sản phẩm — số call tăng theo N product hiển thị.
Giải pháp: Cache exchange rate ngắn hạn theo cặp tiền (ví dụ USD→VND). Nhiều product dùng chung một tỷ giá — giảm call lặp lại.
B2 Catalog Search Full Scan
Vấn đề: Search dùng LOWER() và LIKE %query% — index thường khó phát huy. Latency tăng khi data sản phẩm lớn hơn.
Giải pháp: Thêm LIMIT hợp lý cho search result. Chưa cần OpenSearch — không tăng compute/storage cost.
B3 Checkout Sequential Deps
Vấn đề: Cart, currency, payment, shipping, email, Kafka đều chạy tuần tự. Downstream chậm → latency cộng dồn cho user.
Giải pháp: Async email/post-processing qua Kafka. Idempotency key + outbox pattern cho payment/order — retry an toàn.
Key insight
3 giải pháp đều dùng hạ tầng hiện có: cache trong runtime, LIMIT trong query, async qua Kafka sẵn có. Không thêm hạ tầng mới, không tăng cost — đây là tiêu chí chọn giải pháp trong Week 1.

Khuyến nghị resource và scaling — dựa trên evidence

HPA chỉ triển khai sau khi metrics ổn định 48–72 giờ — không deploy HPA trên nền OOM chưa xử lý.

Bổ sung CPU/memory requests
Nhiều service thiếu CPU requests/limits. Thiếu memory request làm scheduler không đặt pod đúng node. Bổ sung đầy đủ trước khi right-size.
accounting: tăng 200–256Mi
Limit 120Mi gây OOMKilled liên tục — 118 restart / 15h. .NET app với OTel auto-instrumentation + Kafka consumer cần headroom lớn hơn. Verify restart count về 0 trong 24h.
checkout: tăng limit 64Mi
Limit 20Mi, GOMEMLIMIT=16MiB, headroom ~4Mi là risk estimate từ config. Tăng lên 64Mi, đặt GOMEMLIMIT ≈ 51MiB (~80% limit). Verify qua Grafana.
Key insight
Đặt HPA cho frontend và checkout sau khi Prometheus/Grafana metrics ổn định đủ 48–72 giờ. Theo dõi p95/p99 latency và OOM/restart count sau mỗi lần thay đổi resource.

Hai quyết định cost tuần này: NAT Gateway và AWS Budget

Cả hai là conscious trade-off được document rõ — không phải thiếu sót.

🔀 Single NAT Gateway
Lợi ích: Giảm fixed NAT cost so với NAT per AZ. Phù hợp cho giai đoạn baseline và kiểm soát chi phí Week 1.
Trade-off: Chưa đạt network HA tối đa cho outbound. Nếu AZ liên quan gặp vấn đề, outbound từ private subnet bị ảnh hưởng.
Hướng sau: NAT per AZ nếu cần production-grade HA. Theo dõi NAT data processing ($0.045/GB) qua Cost Explorer khi traffic tăng.
🛡 AWS Budget Guardrail (Terraform)
Đã setup: Budget type COST, time unit MONTHLY, default limit $300/month. Threshold notifications: 70%, 90%, 100%.
Security: Email recipients từ GitHub Secret — không hard-code trong repo. Tích hợp vào CI/Terraform plan/apply workflow.
Gap cần xử lý: $300/month vs $300/week target trong tài liệu — cần chốt policy và ghi rõ trong README/evidence.
Key insight
AWS Budgets không ngăn chi phí — nó cảnh báo qua email. Team vẫn cần theo dõi Cost Explorer để so sánh estimate với actual daily/weekly burn rate và validate sau mỗi Terraform apply.

Quick win và những thứ không được cắt trong Week 1

Quick win an toàn nhất: disable LOCUST_AUTOSTART — giảm nhiễu metrics và indirect observability cost ngay lập tức.

✅ Thực hiện ngay
  • Set LOCUST_AUTOSTART=false — tắt synthetic traffic 24/7, metrics phản ánh đúng user traffic thật
  • Validate AWS Budget qua AWS CLI/console sau Terraform apply
  • Set CloudWatch log retention cho log groups không critical
  • Thêm ECR lifecycle policy — tránh image storage tăng âm thầm
  • Chốt budget policy: $300/month hay monthly equivalent của $300/week
  • Thu thập Cost Explorer actual billing data
🚫 Không làm trong Week 1
  • Không giảm Jaeger/Grafana/Prometheus/OpenSearch memory — mất observability là mất evidence
  • Không giảm tài nguyên khi accounting vẫn đang CrashLoopBackOff
  • Không right-size compute khi chưa có 48–72h runtime data ổn định
  • Không tắt observability stack để tiết kiệm — đây là indirect cost cần thiết, không phải lãng phí
Key insight
load-generator autostart tạo synthetic traffic 24/7 → làm nhiễu metrics, tăng trace/log volume, tạo áp lực lên observability stack. Disable trước khi thu thập 48–72h baseline data sạch.

06 · Lỗ hổng

5 lỗ hổng Performance đã xác định

Ưu tiên: fix accounting OOM trước → checkout startup race → checkout memory buffer → checkout async → catalog LIMIT.

P1 accounting OOMKilled
Restart 118 lần / 15h. Memory limit 120Mi không đủ cho .NET app với OTel + Kafka consumer. Fix: tăng lên 200–256Mi, verify restart count về 0 trong 24h qua Grafana.
P2 checkout startup race
Kafka broker chưa sẵn sàng khi checkout khởi động — log báo connection refused, restart 4 lần. Fix: thêm retry/backoff khi tạo Kafka producer lúc startup.
P3 checkout memory buffer
Limit 20Mi, GOMEMLIMIT=16MiB, headroom ~4Mi — risk estimate từ config, chưa phải confirmed OOM event. Fix: limit 64Mi, GOMEMLIMIT ≈ 80% limit.
P4 checkout sequential deps
Latency cộng dồn qua 7+ service. Payment/order nguy cơ double charge khi retry. Fix: async email qua Kafka, idempotency key + outbox pattern cho payment/order.
P5 catalog search full scan
LOWER() + LIKE %query% khó dùng index — latency tăng khi data sản phẩm tăng. Fix: thêm LIMIT hợp lý cho search result trước khi cân nhắc OpenSearch.
Backlog Performance — 9 items tổng
Ngoài 5 lỗ hổng trên: chuẩn hóa Prometheus/Grafana dashboard làm source of truth · N+1 currency cache · HPA sau khi metrics ổn định 48–72h · theo dõi p95/p99 + OOM/restart sau mỗi thay đổi.

5 lỗ hổng Cost đã xác định

Ưu tiên: chốt budget policy → disable LOCUST_AUTOSTART → capture Cost Explorer → CloudWatch + ECR lifecycle.

C1 Cost Explorer chưa đầy đủ
Baseline vẫn là estimate — chưa thấy rõ NAT data processing, ALB LCU, CloudWatch ingestion actual. Fix: chờ billing cập nhật, capture actual daily/weekly burn rate, so sánh với estimate.
C2 Budget unit chưa thống nhất
$300/week trong tài liệu vs $300/month trong Terraform guardrail. Reviewer có thể hiểu sai. Fix: chốt policy rõ ràng, ghi trong README/evidence, điều chỉnh Terraform limit nếu cần.
C3 Load generator autostart
Synthetic traffic 24/7 làm nhiễu metrics, tăng log/trace volume và indirect cost. Fix: set LOCUST_AUTOSTART=false, verify Grafana/Jaeger request volume giảm sau khi tắt.
C4 Observability OOM risk
Jaeger/Grafana có OOM/restart risk. Mất observability = mất khả năng right-size đúng. Fix: không giảm memory, theo dõi OOMKilled/restart count sau khi tắt synthetic traffic.
C5 CloudWatch & ECR lifecycle
Log storage (8 log groups) và image storage có thể tăng âm thầm mà không có alert. Fix: set retention cho non-critical log groups; thêm lifecycle policy cho untagged/dev images trong ECR.

Ngoài 5 lỗ hổng trên: validate AWS Budget qua AWS CLI/console · so sánh estimate vs actual · theo dõi NAT data processing và ALB LCU · compute right-size chỉ sau khi không còn OOM/restart bất thường và có 48–72h data.
