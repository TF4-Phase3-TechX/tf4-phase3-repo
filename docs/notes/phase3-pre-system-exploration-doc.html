1. Mục đích tài liệu & Tóm tắt hệ thống

Tài liệu này được biên soạn nhằm cung cấp cho đội ngũ tiếp quản (TF) một cái nhìn toàn cảnh, sâu sắc và thực tế về nền tảng thương mại điện tử TechX Corp Platform chạy trên Kubernetes trước khi bước vào Phase 3. Mọi thông tin ở đây được tổng hợp trực tiếp từ mã nguồn và các tệp cấu hình có sẵn trong hệ thống.

Mục tiêu chính của đợt khảo sát

Nhận diện cấu trúc
Liệt kê tất cả các microservices hiện hữu, giao thức truyền thông liên kết (gRPC, HTTP, Kafka) và kiến trúc dữ liệu nền tảng.
Bảo vệ SLO
Phân tích sâu luồng nghiệp vụ quan trọng (đặc biệt là checkout path) để khoanh vùng và tối ưu các điểm nghẽn ảnh hưởng SLO doanh thu.
Đánh giá rủi ro
Lập sổ đăng ký rủi ro (Risk Register) từ cấu hình baseline hiện tại (replicas: 1, in-cluster DBs) để chủ động phòng ngừa.
Thiết lập hành động
Đưa ra kế hoạch Tuần 1 rõ ràng với 10 nhiệm vụ ưu tiên (P0/P1) nhằm đưa baseline chạy ổn định và thu thập dữ liệu kiểm chứng.

2.1 Các luồng nghiệp vụ cốt lõi và SLO

TechX Corp Platform là hệ thống đa dịch vụ phức tạp. Việc phân loại luồng giúp phân phối tài nguyên giám sát và tối ưu hóa hợp lý:

Luồng nghiệp vụ Đường đi chính Ý nghĩa vận hành Hạn chế / Drawbacks
Duyệt và tìm sản phẩm frontend → product-catalog → PostgreSQL Trải nghiệm cửa hàng và khả năng tìm kiếm sản phẩm của khách hàng. Truy vấn PostgreSQL dùng LOWER LIKE đồng bộ, dễ chậm khi dữ liệu phình to.
Xem review và hỏi AI frontend → product-reviews → PostgreSQL / LLM Tính năng AI trọng tâm trên trang chi tiết sản phẩm giúp tăng tương tác. Phụ thuộc hoàn toàn vào latency và chi phí tích hợp OpenAI API bên thứ ba.
Giỏ hàng (Cart) frontend / checkout → cart → Valkey Lưu giữ trạng thái mua hàng hiện tại của khách hàng. Valkey chạy in-cluster single-replica, có rủi ro mất giỏ hàng khi pod restart.
Đặt hàng (Checkout) frontend → checkout → cart, catalog, currency, shipping, payment, email → Kafka Luồng tạo ra doanh thu trực tiếp cho TechX Corp (SLO ≥ 99,0%). Luồng đồng bộ quá dài (chain dependencies), một node lỗi làm hỏng cả chuỗi.
Xử lý sau đặt hàng Kafka → accounting, fraud-detection Ghi nhận sổ sách kế toán và phát hiện rủi ro đơn hàng bất đồng bộ. Phụ thuộc vào Kafka queue, nếu Kafka lag sẽ làm chậm tiến trình audit kế toán.

2.2 Những thành phần cần chú ý ngay

Dựa trên cấu trúc hệ thống, đội ngũ vận hành cần đặc biệt lưu ý và giám sát chặt chẽ các thành phần cốt lõi sau:

  • frontend-proxy và frontend: Là entrypoint tiếp nhận toàn bộ request từ internet của khách hàng.
  • Checkout path: Gồm checkout, payment, cart, product-catalog, shipping và currency cấu thành luồng đồng bộ tạo doanh thu.
  • Hạ tầng dữ liệu: PostgreSQL, Valkey và Kafka là các dependency nền tảng quyết định độ bền vững thông tin và sự kiện.
  • Observability stack: OTel Collector, Prometheus, Jaeger, OpenSearch và Grafana là nguồn đo SLO và phục vụ incident investigation (điều tra sự cố).
  • flagd và hook OpenFeature: Cấu thành cơ chế fault-injection do Ban tổ chức quản lý; tuyệt đối không được gỡ bỏ, tắt hoặc đổi hướng.

2.3 Nhận định rủi ro ban đầu (Baseline)

Khảo sát tĩnh (static code analysis) cho thấy cấu hình mặc định chứa nhiều điểm yếu chí tử cần củng cố gấp:

Khu vực đánh giá Hiện trạng baseline Tác động cần theo dõi Drawbacks / Điểm hạn chế
Checkout Path Gọi chuỗi dependency đồng bộ dài liên tiếp Một thành phần lỗi kéo theo checkout thất bại Thiếu timeout ngắn, retry thông minh và cơ chế fallback an toàn.
Availability Cấu hình mặc định replicas: 1, không HPA/PDB Pod, node bị lỗi hoặc quá tải sẽ gây ngắt quãng dịch vụ Không có tính dự phòng cao (High Availability) cho các node quan trọng.
Data Layer PostgreSQL, Valkey, Kafka chạy single-replica in-cluster Tạo ra Single Point of Failure (SPOF) cho toàn hệ thống dữ liệu Không có cơ chế failover tự động và rủi ro mất mát dữ liệu cao.
Observability Prometheus, OpenSearch, Jaeger chưa cấu hình persistence Restart pod observability sẽ làm mất sạch dữ liệu lịch sử sự cố Gây khó khăn lớn cho việc điều tra nguyên nhân gốc rễ (Root Cause Analysis).
AI Reviews Mặc định dùng Mock LLM; có flagd tạo rate limit/accuracy fault Chưa đánh giá được chất lượng, latency và chi phí model thật Chưa có eval dataset chuẩn và fallback khi LLM sập.
Security Grafana anonymous Admin, OpenSearch tắt security plugin Bị xâm nhập thay đổi cấu hình alert hoặc rò rỉ thông tin log Thiếu phân quyền và phơi bày giao diện quản trị ra internet.
Checkout là luồng nghiệp vụ quan trọng nhất tạo ra doanh thu trực tiếp; mọi nỗ lực củng cố hệ thống trong Phase 3 phải ưu tiên bảo vệ luồng này với mục tiêu SLO checkout ≥ 99,0%.

4.4 Dữ liệu và luồng sự kiện

Cấu hình baseline quy định các dịch vụ dữ liệu hoạt động như các dependency nền tảng:

Thành phần dữ liệu Vai trò / Dùng cho Chỉ số runtime cần theo dõi Drawbacks / Điểm yếu baseline
PostgreSQL Lưu catalog sản phẩm, reviews khách hàng và order ghi sổ accounting. Connection pool saturation, query latency, active connections. In-cluster single instance, cạn connection pool sẽ làm nghẽn toàn bộ catalog/checkout.
Valkey Lưu trữ tạm thời trạng thái giỏ hàng (Cart) của khách hàng. Memory usage, read/write latency, restart count, eviction. Chưa bật persistence mặc định, pod restart là bay sạch giỏ hàng của khách.
Kafka Publisher-Subscriber chuyển giao event đơn hàng PlaceOrder bất đồng bộ. Producer send errors, consumer group lag, topic throughput. Chạy Kraft single broker, có rủi ro nghẽn queue và thất thoát tin nhắn nếu broker crash.
OpenSearch Lưu trữ và phân tích logs tập trung của toàn microservices. Disk space utilization, retention period, write latency. Tắt security plugin trong baseline; cardial log phình to gây tràn đĩa in-cluster.
Prometheus Lưu trữ và truy vấn metrics dạng time-series để đo SLO. Scrape time, tsdb disk size, retention, cardinality. Chưa có persistence volume, restart pod sẽ mất sạch biểu đồ xu hướng SLO cũ.
Jaeger Distributed trace phục vụ incident investigation (điều tra sự cố). Sampling rate, trace storage utilization, query speed. Mặc định lưu memory, restart pod là mất lịch sử traces điều tra sự cố.
Mọi dependency trên checkout path đều đồng bộ và có tính quyết định; cần timeout ngắn và fallback an toàn.

4. Danh mục Dịch vụ (Service Directory)

Bảng danh mục dưới đây tổng hợp chi tiết vai trò, các dependency trực tiếp và rủi ro vận hành cần lưu ý đối với toàn bộ 17 service cấu thành hệ thống:

Tên Service Nhóm & Chức năng chính Dependency quan trọng Lưu ý vận hành / Rủi ro chính
frontend-proxy Storefront Ingress Router (Envoy) frontend, Grafana, Jaeger, flagd Cổng vào duy nhất của người dùng; cần xác minh public exposure thực tế trên AWS.
frontend Storefront Next.js App Tất cả các backend gRPC Đóng vai trò tổng hợp UI; là điểm đo lường SLO latency tốt nhất từ góc nhìn khách hàng.
product-catalog Quản lý danh mục sản phẩm PostgreSQL, flagd Tính năng tìm kiếm dùng LOWER LIKE, dễ gây nghẽn PostgreSQL khi lượng sản phẩm lớn.
product-reviews Quản lý reviews & AI Assistant PostgreSQL, LLM, catalog, flagd Gọi LLM API ngoài, cần theo dõi latency, cost per token và xử lý rate limit (429).
llm Mock OpenAI service flagd Là endpoint giả lập phục vụ baseline; cần thay bằng API key thật qua Kubernetes Secret.
cart Thao tác giỏ hàng Valkey, flagd Có sẵn cơ chế retry reconnect Valkey trong code nhưng cart vẫn phụ thuộc vào độ bền dữ liệu Valkey.
checkout Điều phối thanh toán & đặt hàng Các gRPC backend + Kafka Critical Path! Cần giám sát chặt chẽ traces, logs và thiết lập alert lập tức khi lỗi.
payment Thanh toán thẻ giả lập flagd Sự cố INC-3 từng xảy ra do service khởi động chậm và thiếu cấu hình probe chính xác.
shipping / quote Báo giá ship và tạo vận đơn quote service Nằm trên checkout path, một lỗi phát sinh từ shipping/quote sẽ làm checkout thất bại.
currency Quy đổi đơn giá ngoại tệ Không có Service xử lý đồng bộ và không có cache cục bộ; cần HA để tránh single replica crash.
email Gửi email xác nhận đặt hàng flagd Có fault-injection flag giả lập rò rỉ bộ nhớ (memory-leak), cần theo dõi kỹ RAM pod.
recommendation Gợi ý sản phẩm tại trang chủ product-catalog, flagd Không chặn luồng mua hàng chính; cần kiểm tra cơ chế cache và fallback khi catalog lỗi.
ad Hiển thị quảng cáo ngữ cảnh flagd Có fault-injection tăng tải CPU/GC/failure và lag quảng cáo.
accounting Consume đơn hàng để ghi sổ DB Kafka, PostgreSQL Xử lý bất đồng bộ; cần đảm bảo dữ liệu đơn hàng được ghi nhận và persist bền vững vào Postgres.
fraud-detection Consume đơn phát hiện gian lận Kafka, flagd Vị trí thích hợp nhất để đo lường consumer lag và alert khi Kafka bị tồn đọng tin nhắn.
load-generator Locust sinh tải giả lập frontend-proxy, flagd Dùng để kiểm thử tải baseline hệ thống và xác thực tính ổn định sau khi củng cố.
flagd Hạ tầng quản lý feature flags Central flag source của BTC Được bảo vệ bởi BTC, cấm vô hiệu hóa hoặc can thiệp cục bộ để né tránh sự cố giả lập.

5. Hạ tầng Kubernetes & Deployment Baseline

Cấu hình Helm chart mặc định của TechX Corp Platform chứa nhiều điểm chưa tối ưu về mặt High Availability (HA) và phân bổ tài nguyên. Đây là các khoảng trống kỹ thuật lớn cần khảo sát kỹ tại môi trường runtime.

6.1 Quy trình triển khai và kiểm thử baseline

1
Xác thực môi trường
Kiểm tra AWS credentials, context Kubernetes, Helm chart, Docker buildx và quyền đăng nhập ECR registry của đội thi.
2
Build & Push Images
Build toàn bộ 17 container images của platform từ source code và push vào tài khoản ECR được chỉ định của TF.
3
Helm Dependency Build
Chạy lệnh build Helm subcharts cục bộ (Prometheus, OpenSearch, Jaeger subcharts) để sẵn sàng cho nâng cấp release.
4
Deploy Helm Release
Deploy chart với ECR repository của TF, kèm overlay files-observability và bắt buộc values-flagd-sync.yaml.
5
Xác thực Runtime
Kiểm tra pod status, thực hiện port-forward frontend-proxy, đặt đơn hàng thử nghiệm và kiểm tra traces/metrics trên Grafana.

6.2 Đánh giá cấu hình Helm chart mặc định

Hạng mục hạ tầng Thiết lập mặc định từ Helm Values Công việc xác minh tại runtime Rủi ro tiềm ẩn & Hạn chế
Replica Count replicas: 1 cho tất cả các service Kiểm tra số pod thực tế chạy sau khi deploy Helm chart. Không có dự phòng; bất kỳ pod nào crash cũng làm gián đoạn luồng nghiệp vụ.
Autoscaling (HPA) Chưa được bật, không có file manifest HPA/PDB Xác minh xem có HPA được tiêm tự động từ bên ngoài không. Hệ thống không thể tự co giãn khi Locust tăng tải, dẫn đến nghẽn CPU/RAM.
Liveness & Readiness Probes Có template hỗ trợ nhưng values rỗng hoặc thiếu probes Xác minh config probes thực tế của payment, checkout và catalog. Pod mới khởi động chưa kết nối được DB đã nhận traffic, gây lỗi 5xx cho khách (INC-3).
Resource Requests & Limits Có cấu hình limit bộ nhớ nhưng requests thiết lập tùy tiện Đo lượng CPU/Memory sử dụng thực tế khi Locust chạy tải cực đại. Pod dễ bị rơi vào QoS Burstable/BestEffort, dẫn đến bị Kubernetes trục xuất (Eviction).
Node Scheduling & AZs nodeSelector, affinity, và tolerations mặc định rỗng Kiểm tra xem pod được phân bố trên những Node Group và AZ nào. Nhiều pod quan trọng nằm chung một node vật lý, sập node vật lý sẽ sập toàn storefront.
Ingress & Network Boundary Template có hỗ trợ nhưng values mặc định chưa bật Ingress Xác định xem frontend-proxy được expose ra ngoài bằng LoadBalancer hay Ingress. Rủi ro phơi bày trực tiếp Grafana, OpenSearch, Jaeger và DB ra internet công cộng.

6.3 Các Dependency chạy in-cluster dạng Single Replica

Hạ tầng Dữ liệu Thiết lập mặc định Rủi ro vận hành chính Giải pháp củng cố đề xuất
PostgreSQL Deployment in-cluster, 1 replica, PVC đơn lẻ Mất pod Postgres gây sập catalog/accounting; cạn kết nối connection pool (INC-1). Tuning connection pool tối đa, phân tích query log, cân nhắc cấu hình Persistence rõ ràng.
Valkey Deployment in-cluster, 1 replica Valkey restart hoặc sập node sẽ làm mất sạch giỏ hàng của khách hàng (INC-2). Bật persistence (AOF/RDB), cấu hình Persistent Volume để lưu dữ liệu giỏ hàng qua restarts.
Kafka Broker KRaft single broker, 1 replica in-cluster Nếu broker sập, order event sẽ bị mất mát, accounting/fraud-detection bị ngắt kết nối. Bật auto-recovery cho consumers; theo dõi consumer lag và alert kịp thời.
flagd In-cluster pod sync read-only từ central BTC flagd sync lỗi khiến app không đọc được cấu hình sự cố, vi phạm thể lệ cuộc thi. Đảm bảo deploy đúng overlay values-flagd-sync.yaml bắt buộc; không sửa local flags.
Observability Stack Prometheus/OpenSearch/Jaeger cài qua Helm subchart Dữ liệu logs/traces phình to làm cạn kiệt ổ đĩa in-cluster; mất metric khi restart pod. Cân nhắc cấu hình PVC cho Prometheus và OpenSearch, thiết lập retention policy hợp lý.

  • Hệ thống baseline mặc định dùng service llm mock chạy local in-cluster. Để kết nối LLM thật (OpenAI/Anthropic), cần tiêm API keys thông qua Kubernetes Secret và sử dụng overlay values phù hợp.

8.2 Rủi ro của tính năng AI & Biện pháp kiểm chứng

Rủi ro vận hành Cách thức tác động hệ thống Cách thức kiểm chứng & Giảm thiểu Drawbacks / Đánh đổi
Hallucination (Ảo tưởng) Tóm tắt sai lệch review, tự chế thông số sản phẩm không tồn tại. Xây dựng eval set từ reviews thực tế; so sánh đối chiếu facts sản phẩm tĩnh. Đòi hỏi công sức dán nhãn thủ công và xây dựng bộ dữ liệu đánh giá ban đầu.
Rate Limit (429) & Outage Cổng LLM sập kéo theo tính năng xem review của khách bị lỗi/sập. Bật cờ fault-injection của flagd để kiểm thử timeout, retry và fallback an toàn. Chuyển sang chế độ fallback có thể làm giảm trải nghiệm cá nhân hóa của khách hàng.
High Latency (Trễ cao) LLM phản hồi chậm làm nghẽn gRPC reviews, kéo dài thời gian tải trang. Tách riêng latency LLM trong distributed trace; áp dụng streaming hoặc model nhỏ. Model nhỏ hơn (ví dụ: gpt-4o-mini) có thể cho chất lượng tóm tắt kém hơn.
Cost Spike (Chi phí vọt) Người dùng spam câu hỏi dài làm tăng vọt chi phí token của hệ thống. Ghi nhận token usage/cost per request; giới hạn ký tự nhập; cache câu trả lời chung. Cache câu trả lời làm giảm tính cập nhật khi có review mới được thêm vào.
Data Leakage (Rò rỉ dữ liệu) Gửi prompt chứa thông tin nhạy cảm hoặc PII của khách hàng lên LLM. Review biến OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT trước khi dùng dữ liệu thật; ẩn thông tin nhạy cảm trước khi gửi API. Gây khó khăn cho việc debug chi tiết lỗi cụ thể của từng khách hàng.
Quản lý Secret API key LLM bị lộ qua logs, hard-code trong source hoặc không tồn tại khi cần. Xác minh Kubernetes Secret tồn tại và đúng tên; quét logs/traces xem có in ra key không; đảm bảo không hard-code key trong source code. Rotate key gây downtime ngắn nếu không có cơ chế hot-reload; phải kiểm soát RBAC truy cập Secret.
AI Reviews là tính năng bổ trợ (best-effort); tuyệt đối không để lỗi kết nối hoặc trễ của LLM làm đứt gãy luồng xem reviews sản phẩm truyền thống của khách hàng.

8.1 Bảng đánh giá theo 5 trụ cột CDO

Trụ cột CDO Điểm mạnh hiện tại Rủi ro / Khoảng trống Ưu tiên hành động trong Tuần 1
Security (Bảo mật) Một số service chạy non-root; API key LLM dùng K8s Secret. Grafana anonymous Admin, OpenSearch tắt auth, thiếu NetworkPolicy và IRSA. Xác minh public exposure của Grafana; quét log xem có leak dữ liệu nhạy cảm không.

Performance (Hiệu năng) Tích hợp sẵn OTel spans, RED metrics và công cụ Locust. Chưa right-size tài nguyên; search query chậm; không có autoscaling baseline. Tạo baseline load test bằng Locust; trace bottleneck; đo CPU/Memory thực tế.
Cost (Tối ưu chi phí) Cấu hình sẵn ResourceQuota trong thư mục deploy/ để tránh lãng phí. Chưa có AWS Budget/Alert; chưa rõ chi phí NAT Gateway, LB và EBS Disk. Bật AWS Budgets và Cost Anomaly Detection; ghi chép chi tiêu daily spend.

8.2 Lịch sử các sự cố hệ thống cũ

Sự cố lịch sử Nguyên nhân gốc rễ Biện pháp phòng ngừa chủ động Hạn chế / Chi phí đánh đổi
INC-1: Checkout chậm/lỗi do cạn DB connection Dịch vụ catalog và reviews kết nối PostgreSQL không giải phóng pool. Tuning connection pool tối đa; cấu hình alert saturation DB; test tải baseline. Đòi hỏi phải chạy test tải kỹ để tránh đặt giới hạn connection quá thấp gây nghẽn app.

9. Cơ chế Fault Injection (flagd)

Hạ tầng flagd và các OpenFeature hooks được Ban tổ chức (BTC) kiểm soát chặt chẽ để thử thách khả năng tự phục hồi của ứng dụng. TF bắt buộc phải duy trì kết nối này và không được tìm cách vô hiệu hóa.

Luật chơi nghiêm ngặt (10.1): Khi deploy ứng dụng lên EKS, bắt buộc phải kèm theo tệp cấu hình sync phase3/deploy/values-flagd-sync.yaml để flagd nhận cờ sự cố từ central source của BTC ở chế độ read-only. Mọi hành vi gỡ bỏ, tắt feature flag, đổi endpoint sync hoặc bypass cơ chế này đều bị coi là vi phạm luật chơi và dẫn đến disqualification.

12. Đề xuất Backlog ban đầu

Danh sách backlog đề xuất dưới đây được thiết lập dựa trên các khoảng trống kỹ thuật của baseline và bài học sự cố cũ. Thứ tự thực hiện thực tế sẽ do TF quyết định dựa trên mức độ ROI và trần chi phí.

Item cải tiến đề xuất Lý do ưu tiên & ROI Điều kiện tiên quyết trước khi làm Phương án Rollback an toàn
Chuẩn hóa probes cho revenue path Ngăn ngừa sự cố INC-3 khi deploy pod mới; chart đã hỗ trợ probes sẵn. Xác định health endpoint của payment/checkout service. Revert Helm values về cấu hình cũ gần nhất.
Dashboard và alert checkout SLO Không đo lường thì không thể bảo vệ; checkout là revenue-critical. Xác nhận Prometheus metrics và labels thực tế tại runtime. Revert các cấu hình dashboard/rules.
Load test và trace bottleneck Thiết lập baseline tải trước khi tối ưu hiệu năng/chi phí. Thiết lập guardrail tải để tránh làm sập cluster test. Dừng ngay lập tức Locust load generator.
Right-size requests/limits Giảm thiểu lãng phí chi phí EC2 và phòng tránh OOMKilled. Có Prometheus/Grafana resource evidence dưới nền tải thực tế. Revert Helm values cấu hình tài nguyên.
Cấu hình HPA cho storefront/checkout Đảm bảo tự động co giãn pod khi chịu tải Locust lớn. Cấu hình requests tài nguyên ổn định; Prometheus/Grafana evidence ổn định. Xóa HPA manifest hoặc scale replica thủ công về baseline.
Đánh giá Valkey HA & Durability Khắc phục triệt để rủi ro mất giỏ hàng khi node rescheduling (INC-2). So sánh chi phí Persistent Volume với managed Valkey. Trở về baseline, restore dữ liệu giỏ hàng từ snapshot gần nhất.
Kafka lag monitoring & Alerting Bảo vệ luồng accounting và audit ghi sổ đơn hàng. Xác định consumer group và topic metrics chuẩn. Revert dashboard/alert rule Kafka.
Verify Kafka producer ack Phòng tránh lỗi mất đơn hàng bị che giấu trong checkout. Tạo môi trường test lỗi Kafka giả lập và đo latency. Revert cấu hình producer trong mã nguồn/config.
Kiểm soát public exposure Grafana Đảm bảo an toàn bảo mật, tránh leak dữ liệu giám sát hệ thống. Xác minh ingress rules và cấu hình authentication phù hợp. Revert ingress yaml cấu hình bảo mật.
AWS Budgets & Cost Alerting Tránh vượt hạn mức chi phí $300/tuần. Có quyền IAM truy cập AWS Billing. Xóa hoặc điều chỉnh ngưỡng alert trên AWS console.
Báo cáo chi phí theo đơn vị (Unit cost) Cần bằng chứng số liệu cụ thể cho pitch CFO về hiệu quả đầu tư hạ tầng. Có dữ liệu AWS Cost Explorer và số lượng workload thực tế từ Locust. Không áp dụng — đây là báo cáo phân tích, không có rollback.
AI evaluation & Fallback Bảo vệ uy tín thương hiệu khi AI trả lời bậy hoặc LLM bị sập. Có bộ test set eval reviews; UX policy fallback cụ thể. Revert code tích hợp fallback AI.
Kiểm tra Kafka producer acknowledgment Có dấu hiệu event failure có thể bị che khuất — bảo vệ tính toàn vẹn dữ liệu đơn hàng. Reproduce khi Kafka lỗi để quan sát behavior; đánh giá latency trade-off khi dùng acks=all vs acks=1. Revert cấu hình producer trong mã nguồn hoặc Helm values.
Chính sách telemetry GenAI Hạn chế rủi ro rò rỉ nội dung prompt và dữ liệu nhạy cảm qua OpenTelemetry tracing. Xác định loại dữ liệu thực tế đang capture; tắt hoặc mask OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT trước khi export. Env/config rollback — tắt capture làm giảm khả năng debug AI chi tiết.
ADR, runbook, postmortem template Bắt buộc cho Auditability (trụ CDO); chứng minh mọi quyết định có lý do và có người chịu trách nhiệm. Thống nhất ownership trong TF; tạo cấu trúc thư mục docs/adr/, docs/runbook/, docs/postmortem/ và template chuẩn. Không áp dụng — tài liệu không rollback; cần duy trì kỷ luật cập nhật sau mỗi incident.
Tất cả các item cải tiến bắt buộc phải đi kèm phương án rollback chi tiết và được kiểm thử thành công trước khi áp dụng trên môi trường chính.

13. Câu hỏi kiểm chứng & Lệnh vận hành

Các câu hỏi kỹ thuật cần trả lời bằng bằng chứng runtime

Đội ngũ TF bắt buộc phải lấy dữ liệu từ cluster thực tế để trả lời các câu hỏi dưới đây trong tuần đầu tiên:

  • AWS & Cluster: Tài khoản, region, phiên bản EKS, cluster name, namespace và sơ đồ mạng VPC/subnet/NAT/LB thực tế là gì?
  • Storage: Storage class, cấu hình EBS volume, chính sách backup/snapshot của PostgreSQL và Valkey là gì?
  • Public Ingress: frontend-proxy và các giao diện observability (Grafana, Jaeger, OpenSearch) có đang bị expose ra ngoài internet công cộng mà không có mật khẩu hay không?
  • Pod QoS & Probes: Những pod nào trên checkout path đang có replicas > 1, liveness/readiness probes hoạt động và resource requests rõ ràng?
  • Observability Source: Query PromQL và label nào là nguồn dữ liệu chuẩn cho các dashboard tính toán SLO?
  • AWS Budgets: Chi phí hàng ngày phân bổ cụ thể cho EC2, EBS, NAT Gateway, Load Balancer là bao nhiêu? Có gắn tag đầy đủ theo workload?
  • AI Model: Mô hình LLM thật sử dụng nhà cung cấp nào, giá token/triệu và giới hạn rate limit thực tế là bao nhiêu?

Appendix A: Lệnh thu thập thông tin hạ tầng Tuần 1

Dưới đây là các lệnh vận hành thực tế đã được chuẩn hóa để TF chạy trong terminal nhằm thu thập bằng chứng kỹ thuật:

1. Kiểm tra tài khoản AWS, Region và Context Kubernetes 
# Kiểm tra thông tin Identity đang dùng để thao tác AWS
aws sts get-caller-identity

# Kiểm tra AWS Region hiện tại
aws configure get region

# Xem Kubeconfig Context hiện hành
kubectl config current-context

# Kiểm tra thông tin chung của Kubernetes Cluster
kubectl cluster-info

# Kiểm tra phiên bản công cụ Helm và Docker
helm version
docker buildx version
2. Kiểm tra repositories và images trên ECR 
# Liệt kê thông tin repositories trong AWS ECR
aws ecr describe-repositories --region <REGION>

# Xem danh sách các image và tag có trong repository techx-corp
aws ecr list-images --repository-name techx-corp --region <REGION>
3. Deploy ứng dụng với flagd sync của BTC 
# Xây dựng Helm dependencies cho chart cục bộ
helm dependency build ./phase3/techx-corp-chart

# Nâng cấp/Cài đặt Helm Release kèm các tệp sync cờ sự cố bắt buộc của BTC
helm upgrade --install techx-corp ./phase3/techx-corp-chart -n <NS> \
  --create-namespace \
  --set default.image.repository=<REG> \
  -f phase3/deploy/values-observability.yaml \
  -f phase3/deploy/values-flagd-sync.yaml
4. Theo dõi và Giám sát tài nguyên Pods 
# Liệt kê chi tiết pods trong namespace kèm thông tin phân bổ node
kubectl -n <NS> get pods -o wide

# Kiểm tra các tài nguyên deploy, service, hpa, pdb hiện có
kubectl -n <NS> get deploy,svc,hpa,pdb

# Xem chi tiết cấu hình và sự kiện của một pod cụ thể
kubectl -n <NS> describe pod <POD_NAME>

# Xem log truy cập của frontend-proxy để phân tích traffic
kubectl -n <NS> logs deploy/frontend-proxy --tail=100

# Thu thập CPU/Memory tiêu thụ thực tế qua Prometheus/Grafana
# Lưu PromQL result hoặc Grafana dashboard screenshot cho namespace <NS>.

# Port forward để truy cập storefront và các giao diện quản trị nội bộ
kubectl -n <NS> port-forward svc/frontend-proxy 8080:8080

Sau khi thiết lập port-forward thành công, TF có thể truy cập các đường dẫn cục bộ dưới đây để kiểm tra hệ thống:

  • Storefront chính: http://localhost:8080/
  • Giao diện Grafana Dashboard: http://localhost:8080/grafana/
  • Giao diện Jaeger Distributed Trace: http://localhost:8080/jaeger/ui/
  • Giao diện Locust Load Generator: http://localhost:8080/loadgen/
