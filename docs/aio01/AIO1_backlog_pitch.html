AIE why

Vì sao AIE backlog này không over-scope?

Mỗi nhóm task defend được bằng source/requirement evidence: AI hiện tại đang mock, thiếu timeout/fallback/cost/eval và sẽ rủi ro khi bật real LLM.

Gap / risk Evidence hiện tại Task defended Cách nói với mentor
Mock LLM không đo được real quality Baseline trả kết quả giống nhau, latency thấp bất thường, source dùng OpenAI-compatible mock. Eval pipeline, controlled real LLM. “W1 chỉ chứng minh path chạy, W2 mới đo model thật.”
LLM call có thể treo service OpenAI client path chưa có timeout/circuit breaker; gRPC workers hữu hạn. Fallback, timeout, circuit breaker. “Đây là reliability risk, không chỉ AI feature polish.”
Unsafe/misleading output Có flag inaccurate/rate-limit; user/review text đi vào model/tool flow. Guardrail, PII/system prompt filter, faithfulness eval. “Safety task đến từ SLO/AI_FEATURE, không phải nice-to-have.”
Không thấy cost/token/latency Metric hiện tại chủ yếu là AI request counter. AI telemetry fields, cost guard. “Không có telemetry thì không defend được real LLM readiness.”

Incident type Score Data signals Decision

03 · AIOpsReuse framework, adapt RCA
TORAI-lite / BARO-lite

“Lite” nghĩa là bỏ research-heavy parts, giữ phần chạy được

Mục tiêu capstone là detect-correlate-recommend có evidence, không phải reproduce paper benchmark.

TORAI-lite: main AIOps scorer

Giữ ý tưởng multi-source RCA: metric + log + trace + deploy + AI-specific signal.

service_score = 0.35 * metric_anomaly_score + 0.25 * trace_error_latency_score + 0.20 * log_error_score + 0.10 * deploy_correlation_score + 0.10 * ai_specific_signal_score

Bỏ full symptom clustering, causal ranker formal, hypothesis testing đầy đủ.

BARO-lite: metric-heavy latency scorer

Giữ ý tưởng compare baseline window vs incident window để rank service/metric bất thường.

metric_score = abs(incident_mean - baseline_mean) / baseline_std + recent_spike_bonus + slo_metric_bonus

Bỏ full Bayesian online change-point posterior/hazard tuning nếu không đủ thời gian.

Defense: “SOTA algorithms được dùng như reference để nâng scorer, nhưng demo path vẫn deterministic, auditable, và có fallback nếu data không đủ sạch.”

Evidence rule

Definition of Done = evidence-backed

Mỗi task đóng lại phải có artifact review được. Không dùng local-only path làm final evidence trên Jira.

Code / PR evidence

Repo doc link, PR link, commit, command output, test report. Áp dụng cho implementation tasks.

Telemetry evidence

PromQL/OpenSearch query output, Jaeger trace/span link, Grafana screenshot khi cần UI proof.

Eval evidence

Eval report, seed dataset, pass/fail table, limitation note: mock eval vs real LLM quality.

Jira hygiene: Evidence starts local if needed, but final link should point to repo/docs, PR, Jira attachment, or query result. Avoid `E:\...` links in final Jira comment.
